# Прогнозирование полосы частот спиральных структур с помощью машинного обучения (Optuna + MLOps)

## 📌 Описание проекта
Данный проект посвящен применению методов машинного обучения для высокоточного предсказания **полосы частот (bandwidth)** спиральных замедляющих и излучающих систем. 

В радиотехнике и СВЧ-электронике расчет таких характеристик классическими методами (полноволновое моделирование в CST Microwave Studio или ANSYS HFSS) требует колоссальных вычислительных ресурсов и времени (от получаса до нескольких суток на одну конфигурацию). Разработанная ML-модель позволяет проводить мгновенную оценку (инференс за миллисекунды) геометрических и материальных параметров на этапе эскизного проектирования систем.

В проекте реализован полный цикл разработки согласно методологии **MLOps**: от разведочного анализа данных (EDA) до трекинга экспериментов (**MLflow**), автоматического подбора гиперпараметров (**Optuna**) и контейнеризации готового микросервиса (**FastAPI + Docker**).

## 📊 Интеллектуальный подбор гиперпараметров (Optuna + MLflow)
Вместо ручного или случайного перебора параметров в проекте реализован автоматический поиск с помощью фреймворка **Optuna**. Процесс оптимизации полностью интегрирован с **MLflow**: каждый шаг подбора параметров (*trial*) логировался как вложенная сессия (*nested run*) в рамках единого эксперимента `SWS_Helix_Optimization`.
<img width="1880" height="937" alt="mlflow_result" src="https://github.com/user-attachments/assets/8fc21c08-ec52-4969-b3b7-e97a3b8afa90" />

### Пространство поиска параметров:
*   `n_estimators`: от 50 до 300 (с шагом 50)
*   `max_depth`: от 3 до 10
*   `learning_rate`: от 0.01 до 0.3 (логарифмическая шкала)
*   `subsample` и `colsample_bytree`: от 0.6 до 1.0 (контроль переобучения)

### Результаты оптимизации:
Благодаря добавлению фундаментального физического параметра — диэлектрической проницаемости среды (`dielectric_eps`) — и автоматической оптимизации ансамбля деревьев, точность модели выросла в несколько раз по сравнению с базовым решением:
*   **RMSE на тестовой выборке:** `2.2126` — средняя квадратичная ошибка прогноза ширины полосы частот **снизилась в 3.05 раза** относительно Baseline-модели (`6.76`).
*   **Оптимальная архитектура:** Ансамбль из 300 неглубоких деревьев решений (`max_depth: 3`), обученный с применением регуляризации подвыборок строк (`subsample: 0.60`).

## 💡 Физический анализ важности признаков (Feature Importance)
На основе встроенных механизмов оценки XGBoost и зафиксированных в MLflow метрик, был проведен анализ степени влияния факторов на полосу частот:

1.  **`shag_spirali_mm` (Шаг спирали) — ~73.1% важности (Доминирующий фактор)**
    *   *Физическая интерпретация:* Шаг определяет период замедляющей структуры и напрямую задает фазовую скорость волны, а также резонансные свойства системы. Модель абсолютно верно выделила его как главный «рычаг» управления полосой частот.
2.  **`vnutr_diameter_d_mm` (Внутренний диаметр) — ~15.3% важности**
    *   *Физическая интерпретация:* Влияет на волновое сопротивление линии и дисперсионные характеристики, выступая в роли калибровочного параметра.
3.  **`dielectric_eps` (Диэлектрическая проницаемость) и `r_provodnika` (Радиус проводника)**
    *   *Физическая интерпретация:* Включение `dielectric_eps` позволило модели учесть диэлектрическое замедление волны в среде, что радикально уменьшило ошибку предсказания. При этом радиус проводника оказывает минимальное влияние на частотные границы (в реальных СВЧ-устройствах он сильнее связан с омическими потерями и затуханием сигнала).

## 🐳 Развертывание микросервиса в Docker

Для превращения модели в production-ready сервис разработан веб-интерфейс на **FastAPI**. Модель (`xgb_model.json`) и строгий порядок признаков (`model_features.joblib`) импортируются локально, обеспечивая полную автономность контейнера.

### 1. Архитектура репозитория
Перед сборкой убедитесь, что структура файлов выглядит следующим образом:
```text
├── app.py                  # Код веб-сервера FastAPI
├── Dockerfile              # Инструкция сборки контейнера
├── requirements.txt        # Минимальный набор зависимостей
├── xgb_model.json          # Экспортированная модель XGBoost
└── model_features.joblib   # Сохраненный список и порядок колонок
```

### 2. Сборка Docker-образа
Убедитесь, что у вас запущен Docker Desktop, откройте терминал в папке проекта и выполните команду:
```bash
docker build -t spiral-bandwidth-api .
```

### 3. Запуск контейнера
Запустите изолированный контейнер в фоновом режиме с пробросом порта `8000`:
```bash
# Перезапуск контейнера в случае существования старой версии
docker rm -f spiral_api_container
docker run -d -p 8000:8000 --name spiral_api_container spiral-bandwidth-api
```

### 4. Тестирование работы (Inference)
После запуска интерактивная документация API (Swagger UI) становится доступна в браузере по адресу:  
👉 **`http://localhost:8000/docs`**

Вы можете протестировать работоспособность, отправив POST-запрос на эндпоинт `/predict` со следующей структурой JSON:
```json
{
  "shag_spirali_mm": 5.5,
  "vnutr_diameter_d_mm": 12.0,
  "r_provodnika": 0.8,
  "dielectric_eps": 2.1
}
```
Микросервис мгновенно вернет ответ с кодом `200 OK` и точным предсказанным значением полосы пропускания для заданной геометрии и среды.

## 🚀 Направления для дальнейшего развития (Next Steps)
Текущий пайплайн закладывает надежный фундамент автоматизации проектирования. Для дальнейшего развития планируется:
1.  **Feature Engineering:** Переход от абсолютных геометрических размеров к безразмерным физическим соотношениям (например, отношение шага к диаметру $S/D$).
2.  **Добавление прунинга (Pruning):** Использование `optuna.integration.XGBoostPruningCallback` для ранней остановки заведомо неудачных триалов на первых итерациях бустинга с целью экономии времени обучения.
3.  **Кросс-валидация:** Переход на K-Fold кросс-валидацию для исключения переобучения на малых подвыборках инженерных данных.
